BUSINESS CONTEXT

Mueller Auto Part Company was experiencing a compounding visibility crisis. As order volume grew, the gap between operational reality and management awareness widened — not because the data didn't exist, but because it was fragmented across dozens of Excel files with no single source of truth.

This was not a data problem. It was a process transparency problem.

PROBLEMS IDENTIFIED

× Increasing delivery delays, no root cause visibility
× Fragmented Excel tracking across all branches
× No centralised order lifecycle monitoring
× Reactive customer service, zero SLA targets
× No branch-level performance benchmarking
× Revenue impact of delays invisible to leadership

ARCHITECTURE: ON-PREM SQL SERVER

Phase 1 built a SQL Server star schema data warehouse with fact tables for Sales and Order Fulfillment, and dimension tables for Customers, Branches, Products, and Orders.

On-Prem SQL Server→ Star Schema DWH→ Fact: Sales + Fulfillment→ Data Mart

ARCHITECTURE: AZURE CLOUD MIGRATION — MEDALLION

Phase 2 migrated to Azure using a Medallion architecture — Bronze (raw) → Silver (cleaned) → Gold (business layer) — orchestrated end-to-end via Azure Data Factory and Databricks.

On-Prem SQL→ ADF→ Bronze · Raw ADLS Gen2→ Databricks→ Silver · Cleaned→ Gold · Business Layer→ Synapse→ Power BI

THE PROBLEM

Germany's construction SMEs face a dual challenge: government platforms like TED and öffentliche Vergabe publish hundreds of thousands of procurement notices, yet the information is buried in complex nested PDF and eForm documentation. Small firms lack the staff and tools to monitor, filter, and interpret this data — missing opportunities that match their capabilities.

SOLUTION ARCHITECTURE

01
Data Retrieval — TED API + Öffentliche Vergabe
Scheduled API calls to TED (CSV + XML downloads) and Öffentliche Vergabe (eForm ZIP files). Both sources stored raw in AWS S3.
02
Classification & S3 Storage Organisation
AWS Lambda functions classify incoming files by type (PDF, XML, eForm), source, and branch. Structured folder hierarchy in S3.
03
Document Parsing — XML eForms + PDF Extraction
XML eForms parsed into structured JSON. PDF-based TED notices go through text extraction and pattern-based NLP to pull deadlines, CPV codes, contract values, and lot data.
04
Lot-Level Decomposition
Each tender decomposed into individual lots — the atomic bidding unit. Lot-level data normalised across both TED and Öffentliche Vergabe formats.
05
Unified JSON Output + Pipeline Orchestration
All structured data unified into a consistent JSON schema, orchestrated via AWS Step Functions. Lambda handles each stage from raw extraction to structured storage.
06
Dashboard & Filtering Interface
Web-based dashboard allows filtering by CPV code, location, award date, and SME-suitability flags. Live at sme-contract-finder-de.onrender.com.

KEY RESEARCH FINDINGS

LLM APIs (GPT/Claude) would cost €5,000–€15,000/month processing ~300 documents/day. A deterministic rule-based NLP pipeline delivers consistent, auditable extraction at near-zero marginal cost — critical for SME-targeted SaaS.

AWS selected over Azure and GCP based on an 8-criteria evaluation: NLP support, serverless compute, orchestration, storage, notifications, security, cost, and ecosystem maturity.
